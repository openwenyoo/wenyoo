# Architect 设计说明

本文解释 Architect 是为了解决什么问题而设计的，它为什么存在，以及在开发故事或引擎能力时应该如何理解它。

## 一句话概括

Architect 是引擎里的通用世界构建与世界表达智能体：它读取作者编写的世界设定，遵循作者定义的规则与实体，并通过一组基础工具，把玩家意图转化为一致的世界事件、状态变化和面向玩家的叙事输出。

## 设计目的

Architect 要解决的是这样一个问题：

- 玩家会用自由文本和世界交互
- 作者希望用声明式 YAML 定义世界
- 引擎需要一个统一的位置，把作者规则、生成叙事和运行时状态串起来

与其把 LLM 行为分散到许多模块里，Wenyoo 选择使用一个中心化智能体，专门负责解释意图、读取世界状态，并以受控方式提交变化。

## Architect 是什么

Architect 是：

- 一个统一的 LLM 智能体
- 一个运行时世界解释器
- 一个只能通过引擎批准的写工具向玩家表达内容的叙事者
- 一个连接作者数据和玩家实时互动的桥梁

Architect 不是：

- 对作者故事结构的替代品
- 一个可以脱离游戏状态随意写作的自由写手
- 一个会自行发明规则的独立设计层

世界模型的源头仍然是作者写下的 YAML。Architect 的职责是在运行时应用这个模型。

## 理解它的方式

你可以把整个流程想成：

1. 作者定义世界：节点、角色、物品、变量、触发器、表单和规则。
2. 玩家用自然语言或 UI 操作表达意图。
3. Architect 读取相关上下文。
4. Architect 决定世界里究竟发生了什么。
5. Architect 通过引擎工具记录变化，并把结果呈现给玩家。

这意味着 Architect 的工作不只是“写一段文本”，而是先决定世界事件，再把事件表达出来。

## 核心设计思想：先有世界事件，再有叙事

Architect 最关键的设计点，是它以事件为中心，而不是以文案为中心。

在向玩家描述任何事情之前，它应该先确定：

- 发生了什么
- 谁受到了影响
- 哪些状态发生了变化
- 哪些玩家可以感知到这些变化

然后才去叙述这个事件。

这也是为什么整个架构围绕 `commit_world_event` 展开：叙事和状态一起提交，而不是让文本和世界模型逐渐脱节。

## 为什么要做成“通用世界智能体”

Architect 被刻意设计成通用智能体，是为了让引擎能够支持多种题材和交互风格，而不需要把故事特定逻辑硬编码到 Python 里。

这给作者带来了一份很强的契约：

- 用数据定义世界
- 在实体和节点描述里定义规则
- 让 Architect 在具体互动时应用这些规则

这样一来，悬疑故事、奇幻 RPG、生存题材、多人社交叙事都可以使用同一套运行时模式。

## 它如何遵循作者意图

Architect 的设计目标，是优先遵循作者内容，而不是盲目即兴发挥。

运行时它应该按类似下面的优先级理解行为：

1. 实体级规则
2. 节点局部规则
3. 故事整体 lore 和变量
4. 类型和世界常识
5. 最后才是更广义的语言模型推理

实际效果就是：作者决定什么在机制上重要、世界里有哪些实体、什么行为是允许的。Architect 负责把这些规则应用到当前情境，而不是替代作者做决定。

## 依靠工具，而不是裸文本

Architect 的工作方式不是“直接返回任意文本”，而是通过工具完成。

当前设计里最核心的一组运行时工具包括：

- `read_game_state`
- `read_node`
- `commit_world_event`
- `roll_dice`
- `queue_materialization`
- `present_form`

这很重要，因为工具建立了边界：

- 读工具帮助它检查世界
- 写工具确保动作必须通过引擎发生
- 像 `present_form` 这样的 UI 工具，可以在自由文本不够时请求结构化输入

这就是 Architect 如何在受控条件下同时和世界模型、玩家界面交互。

## 动态 UI 也是设计的一部分

Architect 不只是叙述者，它还能驱动动态交互界面。

例如：

- 当玩家需要选职业或回答问卷时，弹出表单
- 在多人模式下为不同玩家发送不同内容
- 把新生成的互动实体加入到游戏状态和 UI 中

这一点非常关键：Architect 是世界交互智能体，而不只是讲故事的工具。

## 叙事与状态的关系

Architect 建立在一条严格原则之上：

- 如果虚构世界里发生了有后果的变化，游戏状态就应该记录它
- 如果游戏状态发生了和玩家相关的变化，玩家能看到的输出就应该反映它

这能避免 LLM 系统里最常见的问题之一：模型说事情发生了，但引擎里没有任何可持续追踪的记录。

在 Wenyoo 里，目标不是“不惜一切代价写得漂亮”，而是先保证世界模拟一致，再让表达自然。

## DSPP 运行时模型

Architect 所面对的是一个分层实体模型：

- `definition`：实体是什么，以及定义它的规则
- `explicit_state`：现在对玩家可见的真实状态
- `implicit_state`：隐藏上下文
- `memory`：角色累计的互动历史
- `properties`：属性、背包、状态、位置等机制数据

这让 Architect 可以清楚地区分：作者定义、可见状态、隐藏状态和机制数据。

## 这个设计为什么对引擎有帮助

这种设计给项目带来几个重要收益：

- LLM 行为集中在一个位置管理
- 减少分散的 prompt 和推理路径
- 让作者规则和运行时行为更一致
- 通过工具扩展能力，而不是加入故事特定 hack
- 更容易支持多人、表单和动态世界变化

## 给开发者的建议

当你添加或修改能力时，尽量保持 Architect 的角色边界：

- 保持核心引擎与具体故事无关
- 优先考虑给 Architect 更好的工具或更清晰的状态
- 避免新增零散的临时 LLM 调用点
- 只要某种玩家可见行为需要长期成立，就为它提供状态表示

如果某个功能改变了 Architect 的推理方式、读状态方式、写事件方式，或它与 UI 的交互方式，请同时更新本文档和 `prompts/architect_system.txt`。
